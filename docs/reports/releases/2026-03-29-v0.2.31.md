# Release v0.2.31 — 2026-03-29

## 1. Tóm Tắt Phát Hành

- **Phiên bản:** 0.2.31
- **Mục tiêu phát hành:** Bổ sung tầng bảo vệ chống spam/DDoS và hoàn thiện bộ trang lỗi HTTP cho hệ thống.
- **Phạm vi thay đổi:** Middleware, settings, templates, env.example — không có thay đổi model/migration.

## 2. Thay Đổi Chính

- **RateLimitMiddleware (mới):** Giới hạn lưu lượng theo IP toàn cục (mặc định 300 req/60 s), đặt ngay sau `SecurityMiddleware` để bảo vệ sớm nhất có thể. Hỗ trợ trả về JSON (API) hoặc HTML trang 429 (browser).
- **UserActivityMiddleware — per-user rate limiting:** Bổ sung giới hạn theo user đã xác thực (mặc định 200 req/60 s) bên cạnh chức năng ghi log hiện có.
- **Bảo mật POST data:** Thêm `_sanitize_post()` mask các field nhạy cảm (`password`, `token`, `api_key`, `csrfmiddlewaretoken`…) trước khi lưu vào `UserActivity`, tránh rò rỉ credentials vào database.
- **Atomic rate limiting:** Chuyển từ pattern `get + incr` sang `incr-first` (atomic trên Redis), loại bỏ race condition trên môi trường multi-worker.
- **WebSocket exclusion:** Thêm `/ws/` vào `exclude_paths` của `UserActivityMiddleware` — WS handshake không bị log và không tính vào per-user rate limit.
- **Cảnh báo startup:** Khi `DEBUG=False` mà không dùng Redis, in WARNING để nhắc nhở rate limit không chính xác trên multi-worker.

## 3. Tính Năng Mới

- **Bộ trang lỗi HTTP đầy đủ:** `400.html`, `403.html`, `404.html`, `429.html`, `500.html` — thiết kế tối giản, dark mode, phù hợp với admin panel.
- **base.html skeleton:** Template cơ sở dành riêng cho trang lỗi, tự chứa (inline CSS, không phụ thuộc file static ngoài), định nghĩa đầy đủ các block: `lang`, `title`, `meta`, `styles`, `extra_head`, `body_attrs`, `body_start`, `header`, `content`, `main_attrs`, `footer`, `body_end`, `extra_scripts`.
- **env.example tái cấu trúc:** Tổ chức theo section (General, Domains & CORS, Database, Cache & Redis, Celery, Email, Rate Limiting, Monitoring, UI/Feature flags), bổ sung `RATE_LIMIT_REQUESTS`, `RATE_LIMIT_WINDOW`, `USER_RATE_LIMIT_REQUESTS`, `USER_RATE_LIMIT_WINDOW`.

## 4. Tương Thích Và Rủi Ro

- **Tương thích:** Không có breaking change. Các biến rate limit có giá trị mặc định an toàn — không cần cấu hình thêm nếu không muốn thay đổi ngưỡng.
- **Yêu cầu Redis cho production:** Rate limiting chính xác cần Redis (`USE_CACHE=True`, `CACHENAME=redis`). Với `LocMemCache`, mỗi worker có bộ đếm riêng — hệ thống vẫn chạy nhưng ngưỡng thực tế là `RATE_LIMIT_REQUESTS × số worker`.
- **Rủi ro trang lỗi:** Django tự động phục vụ 400/403/404/500 qua `APP_DIRS=True` — không cần đăng ký handler. Template 429 được render thủ công bởi middleware. Nếu Redis down và fallback không hoạt động, rate limit bị vô hiệu hóa tạm thời (fail-open, an toàn cho uptime).

## 6. Hướng Dẫn Nâng Cấp

**Cài từ GitHub theo tag:**
```bash
uv add "git+https://github.com/White-Neuron/django-whiteneuron.git@v0.2.31"
```

**Bước triển khai:**
1. Thêm `RateLimitMiddleware` vào `MIDDLEWARE` ngay sau `SecurityMiddleware`.
2. Đảm bảo `UserActivityMiddleware` nằm sau `AuthenticationMiddleware`.
3. Thêm các biến rate limit vào `.env` nếu muốn override mặc định.
4. Cấu hình Redis (`USE_CACHE=True`, `CACHENAME=redis`) cho môi trường production multi-worker.
5. Chạy `python manage.py collectstatic` — các template lỗi không cần static file ngoài.

**Điều kiện rollback:**
- Downgrade về `v0.2.30` không gây mất dữ liệu (không có migration mới).
- Xóa `RateLimitMiddleware` khỏi `MIDDLEWARE` là đủ để vô hiệu hóa rate limiting.
